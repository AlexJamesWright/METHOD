8 /*
9  Class contains all the data of the simulation relevant to any of the other
10  modules. Containing it in this way prevents issues of cyclic includes, also
11  results in Simulation as more of an interface than a class that needs to be
12  known to lower objects---good practice.
13 
14  Usage
15  -----
16  Call the constructor with at least: the number of cells and domain limits in
17  the x, y, and z direction, and the simulation end time. As ghost cells are
18  automatically added to the domain, there would be a minimum number of total cells
19  in any direction of nine---as a result, running 2D simulations would be 9x slower
20  than necessary. To combat this, you can request zero cells in the z-direction
21  which is equivalent to a 2D domain where Nz=1. Keep this behaviour in mind when
22  constructing new models.
23  Other variables, such as the courant factor, can also be set in the constructor
24  but by default have sensible values that should work for most set ups, these can
25  largely be ignored unless the simulation is failing to converge.
26  Selecting a model will automatically set the number of cons prims and aux vars,
27  both inside the model class and in the data class (although it is the data class
28  values that are accessed by the various functions).
29  The elementID function data.id(var, i, j, k) is a useful shortcut for accessing
30  data belonging to a specific cell, where var is the id of the variable in the array
31  we wish to access, and (i, j, k) corresond to the (x, y, z) coordinates of the
32  cell we wish to access. Note: this includes ghost cells, so in practice these values
33  can range from 0 <= (i,j,k) < (nx,ny,nz)+2*Ng
34  or 0 <= (i,j,k) < (Nx,Ny,Nz) assuming we are working in 3D.
35  For 2D simulations, k=0 at all times.
36 */
